程序编写¶

在开发过程中，主要用到的是MapReduce，但是由于一些条件限制，所以要将MR翻来覆去的玩个遍。主要有一下几个事情：

  1. MR串联.
  2. 多文件输入
  3. 多文件输出.
  4. DistributedCache.

MR串联¶

在去年开发Mahout的时候，需要不断的将MR串联，比如对于K-Means算法，需要不断的迭代。那么在这次开发任务中同样遇到来这样的问题，那么我就在本地写个例子。贴出主要的代码，并加以说明，以方便自己记忆。

首先看一下我要处理的数据：

111,you
222,are
333,a
444,bitch
222,you
333,say
111,yes

接着我写了第一个Test类，其中只写了Map，没有用到Reduce。目的是直接输出：

static class DataProcess extends
        Mapper {
    private Text inkey = new Text();

    protected void map(LongWritable key, Text value, Context context)
            throws IOException, InterruptedException {
        String[] words = value.toString().split(
                StringUtils.DELIMITER_INNER_ITEM);
        inkey.set(words[0] + StringUtils.DELIMITER_INNER_ITEM + words[1]);
        context.write(inkey, NullWritable.get());
        System.out.println(inkey.toString());
    }
}

看一下我配置的Job：

public static void firmap(Path input, Path output) throws IOException,
        ClassNotFoundException, InterruptedException {
    Configuration conf = new Configuration();
    Job job = new Job(conf, "UpDownAnalysisJob");
    job.setMapperClass(DataProcess.class);
    job.setNumReduceTasks(0);
    job.setOutputKeyClass(Text.class);
    job.setOutputValueClass(NullWritable.class);
    job.setJarByClass(Test.class);
    // job.setOutputFormatClass(Text.class);
    job.setOutputFormatClass(SequenceFileOutputFormat.class);
    FileInputFormat.addInputPath(job, input);
    // SequenceFileOutputFormat.setOutputPath(job, new Path(args[1]));
    FileOutputFormat.setOutputPath(job, output);
    job.waitForCompletion(true);
}

在其中我用到了SequenceFileOutputFormat。Hadoop的MapReduce不只是可以处理文本信息，它还可以处理二进制格式的数据。它可以快速的序列化任意的数据类型到文件中，而对应SequenceFileInputFormat则会把文件反序列化为相同的类型并提交为下一个Mapper的输入数据，方式和前一个Reducer的生成方式一样。它的优点为：紧凑、快速、可扩展、支持互操作

看一下我的job配置：

public static void secmap(Path input, Path output) throws IOException,
        ClassNotFoundException, InterruptedException {
    Configuration conf = new Configuration();
    Job job = new Job(conf, "maptest2");
    job.setMapperClass(DataProcess2.class);
    job.setNumReduceTasks(0);

    job.setInputFormatClass(SequenceFileInputFormat.class);
    job.setOutputKeyClass(Text.class);
    job.setOutputValueClass(NullWritable.class);
    job.setJarByClass(Test2.class);
    // job.setOutputFormatClass(SequenceFileOutputFormat.class);
    FileInputFormat.addInputPath(job, input);
    // SequenceFileOutputFormat.setOutputPath(job, new Path(args[1]));
    FileOutputFormat.setOutputPath(job, output);
    job.waitForCompletion(true);
}

最后我写了一个API来调度它们：

public static void main(String[] args) throws ClassNotFoundException, IllegalArgumentException, IOException, InterruptedException {
    Path map1Path = new Path(args[1], new Path("map1"));
    Test.firmap(new Path(args[0]), map1Path);
    Path map2Path = new Path(map1Path,new Path("map2"));
    Test2.secmap(map1Path, map2Path);
}

首先，我将定义出第一个MR的输出路径map1Path，接着再定义第二个MR的输出路径，最后我用第一个MR的输出路径作为第二个MR的输入路径来运行程序。最终得到的目录结构为：

多文件输出¶

由于对于不同的结果最终将输出到不同的Hive表中，所以需要实现多个输出。 需要在Reducer里面将方法重写：

 @Override  
 protected void setup(Context context) throws IOException, InterruptedException {  
     mos = new MultipleOutputs(context);  
 }  

 @Override  
 protected void reduce(Text key, Iterable values, Context context) throws IOException, InterruptedException { 
     if (...) {
         mos.write("FileName1", key, value)
     } else if () {
         mos.write("FileName2", key, value)
     }

 }

 @Override  
 protected void cleanup(Context context) throws IOException, InterruptedException {  
     mos.close();  
 }

在job中就需要根据Reducer中的设置与其遥相呼应：

MultipleOutputs.addNamedOutput(job, "FileName1", TextOutputFormat.class,  
        Text.class, LongWritable.class);  

MultipleOutputs.addNamedOutput(job, "FileName2", TextOutputFormat.class,  
        Text.class, LongWritable.class);

总结¶

以上内容就是今天我想要讲的这几天的工作内容之一，这些技术在实施起来非常简单，但是也非常有趣，如果想增加乐趣性，可以通过将上面的几终方法进行组合。

比如我现在要处理3张表,分别为A，B，C，第一步需要将它们进行去重，那么就在Reduce中直接输出可以了，第二步需要将C表的数据分别和A、B表中的数据做比较，由于C表去重之后就已经很小了，所以我可以将C表处理后的数据通过DistributedCache的方法缓存起来，以更高效的去读取数据。

所以，可以看到，根据上面的需求，我同时需要用到今天将的所有内容。首先，我需要多文件输入，然后将MR进行串联，再将C表进行DistributedCache，最后多文件输出。

其实，上面的话说谎来了，打脸....我并没有用到多文件输出，因为最终的结果会成为一张表。

在这里我在多说几句：

多文件输出这种方法并不好，如果非不得已，最好别用，因为这样的做法有很大的缺点。首先在运行job之前就需要知道分区数以分配reducer的数目，但是让程序来严格限定分区数并不好，因为可能导致分区数少或分区不均。让很多reducer做少量的工作巴士一个高效的作业组织方法，比较好的方法是使用更少的reducer做更多的事情。
