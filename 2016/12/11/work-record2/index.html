Hadoop HA搭建（2.5.2）¶

一共有5台服务器,redhat系统，如果对自己机器配置不清楚的同学，可以通过 cat /proc/cpuinfo |more 或 dmesg | grep CPU 命令看一下机器的cup的情况。getconf LONG_BIT命令可以查看cpu的位数；用 free -m 命令看一下机器的内存情况。

这五台机器就不用介绍部署情况了吧，老一套，两台用来做namenode和resourcemanager，实现高可用，三台用来做datanode. 在这里我想说一下，其实做4台datanode也是可以的，反正有一台始终是standby状态，闲着也是闲着，当然要保证namenode和ResourceManager的active状态都运行在同一台机器上，减轻主节点的一些负担嘛。当然这是我个人的一点愚见，想批评教训我的同学请随时发邮件至 sunnyin127@gmail.com .

启动顺序¶

我直接跳过了配置文件以及密钥配置这两个步骤，比较简单，为了节省篇幅，我就直接说说集群的启动顺序吧。

启动Zookeeper集群¶

Zookeeper集群一共配了三台，都装在来datanode节点上。配置文件里的clienPort就默认为2181就行，除非非不得已，要不别乱改了。因为它需要和core-site.xml以及yarn-site.xml文件中的配置保持一致，比较容易忽略。(这是一年前踏过的坑.......)

今天踩到的坑是在我启动zookeeper时其他的机器总是起不来，防火墙肯定是关了的，就是尼玛起不来,报错为ConnectionConfused.最后看了日志惊呆了，只有一句话：找不到java....汗脸...

但是运行java -version发现是存在java的，好了不饶弯了。是因为系统读取的是自带的java环境变量，而非是自己配置的。

修改来系统环境变量竟然没有source一下。

Zookeeper启动之后执行在对应的机器上运行jps命令会看到如下进程：

QuorumPeerMain
Jps

接着在其bin目录下执行./zkCli.sh && ls /

如果除了看到zookeeper或者hadoop-ha成员，还有其他成员存在，想想你是否会用到，不用的话是清理一下，省得之后会麻烦踩坑。

命令为 rmr /member_name

格式化 zookeeper 集群¶

目的是在 zookeeper 集群上建立 HA 的相应节点

命令为 hdfs zkfc -formatZK

启动journalnode集群¶

journalnode我都配置在了datanode上。在官网上原话是这么说的：

The JournalNode daemon is relatively lightweight, so these daemons may reasonably be collocated on machines with other Hadoop daemons, for example NameNodes, the JobTracker, or the YARN ResourceManager. Note: There must be at least 3 JournalNode daemons, since edit log modifications must be written to a majority of JNs. This will allow the system to tolerate the failure of a single machine. You may also run more than 3 JournalNodes, but in order to actually increase the number of failures the system can tolerate, you should run an odd number of JNs, (i.e. 3, 5, 7, etc.). Note that when running with N JournalNodes, the system can tolerate at most (N - 1) / 2 failures and continue to function normally.

翻译：装在什么地方无所谓，轻量级，但至少 3 台机器并且是单数。

启动命令：hadoop-daemon.sh start journalnode （在每台配置journalnode的机器中都启动起来）

一旦启动了JournalNode，必须首先同步两个HA NameNode的磁盘元数据。

在官网中是这么说的：

If you are setting up a fresh HDFS cluster, you should first run the format command (hdfs namenode -format) on one of NameNodes.

If you have already formatted the NameNode, or are converting a non-HA-enabled cluster to be HA-enabled, you should now copy over the contents of your NameNode metadata directories to the other, unformatted NameNode by running the command “hdfs namenode -bootstrapStandby” on the unformatted NameNode. Running this command will also ensure that the JournalNodes (as configured by dfs.namenode.shared.edits.dir) contain sufficient edits transactions to be able to start both NameNodes.

If you are converting a non-HA NameNode to be HA, you should run the command “hdfs namenode -initializeSharedEdits”, which will initialize the JournalNodes with the edits data from the local NameNode edits directories.

翻译一下：

  1. 如果第一次搭建集群，执行此步骤前先格式化namenode

  2. 如果你已经格式化namenode了，或者你想将你之前非高可用的集群换成高可用的，那么你需要将你现有的namenode的元数据拷贝到其它的namenode中。在其上面执行 hdfs namenode -bootstrapStandby命令，它同时也会确认journalnode的配置信息。

  3. 如果你是将非高可用的集群换成高可用的，你需要执行 hdfs namenode -initializeSharedEdits 来初始化数据。

启动namenode¶

首先需要格式化namenode。

格式化的时候需要注意一个问题。

如果你是第一次格式化，那么执行命令之后就会很流畅的结束。如果你并非是第一次格式化，那么你需要查看日志信息中所显示的 clusterID 是否一致。

在格式化了的机器上启动它 hadoop-daemon.sh start namenode

之后去另一台机器上执行 hdfs namenode -bootstrapStandby 用以同步两个namenode的元数据。

启动其上的 namenode hadoop-daemon.sh start namenode

ok，至此你可以看到namenode的进程了。并且在web上通过50070(没有修改的情况下)端口可以看到页面。

但是此时可能会出现一个问题，你会发现两台namenode都处于 standby 状态。那么就需要将其中一台转换为 active 状态。

命令为 hdfs haadmin -transitionToActive

也许会让你加上强转标识，那你就加上.(抱歉因为这篇文章我在家中写下的，没法用hadoop命令，没法贴强转的单词了....)

非得告诉我 datanodeuuid 没有分配，我上网查了一下，结果全都在说clusterID的事情，真TM能扯，某度怎么就能给我匹配出这么个东西呢，跟它有什么关系！

后来在这个坑中吃了顿饭，尿意来临就地泚了一泡，写上 老孙到此一游 4个大字之后，一个筋斗云终于逃出了坑。原来是这么回事。

在 hdfs-site.xml 配置中有这么个参数 dfs.datanode.failed.volumes.tolerated, 它是啥意思嘞？官网上是这么说的：

The number of volumes that are allowed to fail before a datanode stops offering service. By default any volume failure will cause a datanode to shutdown.

就是说这个值表示着在将datanode定义为挂掉之前，我可以失败的卷数。老大的配置中配了个 3 。我艹嘞，牛逼.

在源码中显示，这个值的大小不能超过dfs.datanode.data.dir所设置的盘数，这个值我没有射啊，默认1个吧，我竟然dfs.datanode.failed.volumes.tolerated中设置了3.

之后我索性改为了 0, ok，一切正常。

启动yarn¶

在 rm1 的机器上启动yarn start-yarn.sh ，这个命令同时会将datanode上的nodemanager服务启动起来。

之后在 rm2 机器上单独启动 yarn 服务 yarn-daemon.sh start resourcemanager

启动 ZKFC¶

格式化： hdfs zkfc -formatZK

在每台rm机器上 hadoop-daemon.sh start zkfc 来启动 zkfc.

开启历史日志服务¶

在每台主节点上执行 mr-jobhistory-daemon.sh start historyserver
